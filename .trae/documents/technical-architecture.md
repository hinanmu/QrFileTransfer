# 技术架构文档 - QrFileTransfer

## 1. 技术选型

### 1.1 核心技术栈
| 技术 | 用途 | 说明 |
|------|------|------|
| **HTML5** | 页面结构 | 单页应用 |
| **CSS3** | 界面美化 | 自定义样式，无框架 |
| **Vanilla JavaScript (ES6+)** | 核心逻辑 | 无外部框架依赖 |
| **qrcode** (CDN) | QR 码生成 | 支持 Byte Mode 原生二进制编码，避免 UTF-8 编码膨胀 |
| **jsQR** (CDN) | QR 码解码 | 纯 JS QR 解码器，支持 Byte Mode 解码 |
| **Web File API** | 文件读取 | `FileReader`, `webkitdirectory` |
| **WebRTC/getUserMedia** | 摄像头访问 | 实时视频流捕获 |
| **Canvas 2D API** | 视频帧捕获 | 从视频流捕获帧用于 QR 解码 |

### 1.2 为什么选择原生 JS
- 无需构建工具链，单 HTML 文件即可运行
- 完全离线可用，不依赖构建环境
- 最大化兼容性
- 学习成本最低

## 3. 核心模块设计

### 3.1 发送端模块 (Sender)

#### 3.1.1 FileScanner
- 使用 `<input type="file" webkitdirectory>` 选择文件夹
- 使用 `<input type="file">` 选择单个文件
- 递归遍历文件树，构建文件清单
- 计算总大小、总文件数

#### 3.1.2 EncoderEngine
- 将每个文件按 **2,300 字节** 分片
- 构建数据包（添加协议头）
- 使用 `qrcode.js` 生成 QR 码 Canvas 元素
- 将所有 QR 码存储在队列中
- 关键类：
  ```javascript
  class EncoderEngine {
    constructor()        // 初始化编码器
    addFile(file, path)  // 添加文件到队列
    encodeAll()          // 编码所有文件
    getQRCount()         // 获取 QR 码总数
    getQRCode(index)     // 获取指定索引的 QR 码
    getMetadata()        // 获取元数据 QR 码
  }
  ```

#### 3.1.3 QRPlayer
- 全屏显示 QR 码轮播
- 支持自动播放（默认 **2 秒/帧**，可配置 0.5~5 秒/帧）
- 元数据包（第一个 QR）固定展示 5 秒，给摄像头初始化时间
- 支持暂停/继续
- 支持手动前后翻页
- 进度指示器（当前帧 / 总帧数）
- 轮播速度控制滑块（步进 0.1 秒）

### 3.2 接收端模块 (Receiver)

#### 3.2.1 CameraController
- 通过 `navigator.mediaDevices.getUserMedia` 获取摄像头权限
- 获取视频流并在隐藏的 `<video>` 元素中播放
- 提供视频帧捕获接口
- 处理摄像头切换（前置/后置）

#### 3.2.2 DecoderEngine
- 使用 `jsQR` 库从 Canvas 帧中解码 QR 码
- 解析数据包协议头
- 按文件索引和分片索引排序存储
- 去重处理（同一分片可能被多次扫描到）
- 检测传输完成（所有分片就绪）
- 核心逻辑：
  ```javascript
  class DecoderEngine {
    constructor()                    // 初始化解码器
    decodeFrame(canvas)              // 解码单帧
    processPacket(packet)            // 处理数据包
    isTransferComplete()             // 检查传输是否完成
    getMissingChunks()               // 获取缺失的分片列表
  }
  ```

#### 3.2.3 FileReconstructor
- 根据元数据重建目录结构
- 按顺序拼接分片数据
- 使用 `Blob` 和 `URL.createObjectURL` 生成下载链接
- 批量下载文件夹结构（ZIP 可选，但为保持零依赖，采用逐个下载 + JSZip 可选）

### 3.3 协议层设计

#### 3.3.1 包类型定义
```javascript
const PACKET_TYPE = {
  META_DATA:   0x02,  // 元数据包（文件清单+目录结构），第1个QR码
  FILE_DATA:   0x01,  // 文件数据分片
  TRANSFER_END: 0x03  // 传输结束标记（最后一个QR码）
};
```

#### 3.3.2 文件数据包结构 (0x01)
```
[1 byte ] 包类型 (0x01)
[4 bytes] 文件索引 (uint32, 大端序)
[4 bytes] 分片索引 (uint32, 大端序)
[4 bytes] 总分片数 (uint32, 大端序)
[2 bytes] 路径长度 (uint16, 大端序)
[可变   ] 相对路径 (UTF-8, 如 "docs/subdir/report.txt")
[4 bytes] 本片数据长度 (uint32, 大端序)
[<=2296 ] 分片数据
```

#### 3.3.3 元数据包结构 (0x02)
```
[1 byte ] 包类型 (0x02)
[4 bytes] 文件总数 (uint32, 大端序)
[每个文件重复:]
  [2 bytes] 路径长度 (uint16)
  [可变   ] 相对路径 (UTF-8)
  [8 bytes] 文件大小 (uint64, 大端序)
  [4 bytes] 总分片数 (uint32, 大端序)
```

#### 3.3.4 传输结束包 (0x03)
```
[1 byte] 包类型 (0x03)
[4 bytes] 文件总数 (uint32, 与元数据校验)
[4 bytes] CRC32 校验和
```

### 3.4 加密设计

#### 3.4.1 XOR 流密码
- **算法**：简单 XOR 流密码，密码字节循环与数据逐字节异或
- `encrypted[i] = data[i] ^ password[i % password.length]`
- **解密**：相同密码再做一次 XOR 即可还原
- **密码为空时不加密**，数据原样传输
- 加密发生在分片打包完成后、QR 码生成前
- 解密发生在 QR 码解码后、协议头解析前

### 3.5 数据流

```
发送端:
  文件(File) → ArrayBuffer → Uint8Array → 分片 → 
  添加协议头 → 生成QR码(Canvas) → 轮播显示

接收端:
  摄像头 → 视频帧(Canvas) → ImageData → jsQR解码 →
  提取数据 → 解析协议头 → 按文件/分片索引存储 →
  所有分片到达 → 重组文件 → Blob → 下载
```

## 4. 数据编码详述

### 4.1 QR 码参数
| 参数 | 值 |
|------|-----|
| 版本 | 自动 (最高 40) |
| 纠错级别 | L (7%) |
| 模式 | Byte |
| 最大数据容量 | **2,953 字节** (Version 40, Byte Mode, Level L) |
| 实际载荷上限 | **2,600 字节** (协议头 + 数据) |
| 数据区大小 | **2,600 字节** (每包) |
| 协议头大小 | ~104 字节 (含路径、索引等) |

### 4.2 文件分片算法
```
对于每个文件:
  1. 读取文件为 ArrayBuffer
  2. 转换为 Uint8Array
  3. 计算总分片数 = Math.ceil(文件大小 / 2600)
  4. 按 2600 字节切分
  5. 每片创建数据包:
     [包类型(1) + 文件索引(4) + 分片索引(4) + 总分片数(4) + 
      路径长度(2) + 相对路径(可变) + 本片数据长度(4) + 数据(<=2600)]
  6. XOR 加密（如设置了密码）
  7. 编码为 QR 码（Byte Mode）
```

### 4.3 文件夹结构编码策略
文件夹的**目录树结构**通过两种方式编码，确保接收端能完美还原：

**方式 A：元数据包（先导包）**
- 传输前发送一个元数据 QR 码（索引 0）
- 包含完整文件清单：`文件总数 + 每个文件的[相对路径, 大小, 总分片数]`
- 相对路径保留完整目录结构，如 `"photos/2024/vacation/img001.jpg"`
- 接收端据此预知要接收的所有文件和目录结构

**方式 B：数据包路径冗余**
- 每个文件数据分片也携带完整的相对路径
- 即使元数据包丢失，接收端也能从数据分片中恢复路径信息
- 路径冗余确保传输健壮性

**还原过程：**
1. 接收端解析元数据包 → 获得完整目录树
2. 接收所有数据分片 → 按 (文件索引, 分片索引) 排序
3. 提取每个文件的路径 → 在内存中重建目录树
4. 文件数据拼接完成 → 按路径创建下载结构

### 4.4 传输流程
```
Phase 1: 元数据交换
  Sender → QR码(元数据包) → Receiver 解码成功 → 确认文件清单

Phase 2: 数据分片传输
  Sender → 轮播所有数据QR码 →
  Receiver → 持续扫描 → 实时存储分片

Phase 3: 传输完成
  Sender → 显示结束标记QR码 →
  Receiver → 检测到结束 → 开始重组文件
```

## 5. UI 设计

### 5.1 设计风格
- **主题**：赛博朋克/数据流风格
- **配色**：深色背景 (#0a0a1a)，霓虹青 (#00ffc8) 作为主色调，霓虹紫 (#7b2ffc) 作为辅助色
- **字体**：等宽科技感字体 + 可读性好的无衬线字体
- **动效**：数据流动画，扫描线效果，脉冲光效

### 5.2 布局
- **顶部**：Logo + 模式切换开关
- **发送端视图**：
  - 左侧：文件列表/控制面板
  - 右侧/中央：QR 码轮播区域（大尺寸）
  - 底部：进度条和控制按钮
- **接收端视图**：
  - 中央：摄像头预览（画中画模式）
  - 侧边：接收状态/文件列表
  - 底部：进度和统计信息

### 5.3 交互反馈
- 成功解码时：绿色闪光 + 振动动画
- 遗漏分片时：红色高亮提示
- 传输完成：庆祝动画 + 自动下载

## 6. 限制与边界

### 6.1 已知限制
- 单个 QR 码最大数据量约 2.4KB
- 传输大文件需要大量 QR 码（10MB = ~4,350 个 QR 码）
- 扫描速度受限于摄像头帧率和环境光照
- 需要稳定的摄像头对焦和合适的距离

### 6.2 优化策略
- 使用纠错级别 L 最大化数据容量
- 全屏展示 QR 码以便摄像头识别
- 自动亮度/对比度调整
- 多帧采样去重（同一 QR 码被多次扫描时只保留一次）
- 动态调整轮播速度

## 8. 错误处理
- **扫描超时**：同一 QR 码扫描超时后自动跳转下一个（发送端自动轮播）
- **解码失败**：静默跳过，下一帧重新尝试
- **分片重复**：基于分片索引去重
- **传输中断**：显示已接收的分片列表，支持重新开始
- **摄像头错误**：友好的错误提示和权限请求引导
